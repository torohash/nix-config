# 実装前の要件明確化ルール

目的は、課題を **検証可能な要件（spec）に変換してから** 実装することです。
これはリスクを止めるためのルールではなく、**仕様を明確にするための** ルールです。
基準は「作れそうか」ではなく **「完了を判定できるか（テストで合否が言えるか）」**。
（NASA の要求定義と同じ立場: 要求は検証可能・テスト可能・測定可能であるべき。）

---

## 0. 基本姿勢

課題を受け取ったら即実装しない。まず **その課題が検証可能な要件になっているか** を見る。
なっていなければ、未確定点を次の3モードで潰し、spec を一意に確定させる。

| モード | いつ使うか | Claude Code での動作 |
| --- | --- | --- |
| **Investigate（調べて決める）** | リポジトリを見れば spec が分かる | Read / Grep / Glob で既存実装・仕様・テストから導出する |
| **Assume（仮定して決める）** | 導出はできないが妥当な既定値で埋められる | 仮定を明示・記録して進める（後から直せる形で） |
| **Ask（聞いて決める）** | spec が本質的に分岐し、導出も安全な仮定もできない | `AskUserQuestion` で最小限だけ質問する |

優先順位は **Investigate ＞ Assume ＞ Ask**。
Ask は最も遅い経路。人に聞く前に「調べて決まらないか」「仮定で決まらないか」を先に試す。

---

## 1. 「要件が定義できた」と言える条件

次が **すべて** 言えるなら、実装してよい。

- 成功条件がテスト可能な形で書ける（何ができたら完了かを合否判定できる）
- 対象範囲と非対象範囲が言える
- 主要な入力・出力が言える
- 主要な異常系の期待挙動が言える
- 仮定した点が明示されている

一つでも言えないなら spec は未定義。Investigate → Assume → Ask の順で埋める。

---

## 2. 判定フロー

```text
未確定点を見つけたら、上から順に判定する。

(1) リポジトリ（既存コード・仕様・テスト・規約）を見れば決まるか?
      はい  → Investigate（調べて spec を確定）
      いいえ → 次へ

(2) その点が変わると、成果物（成功条件・挙動・入出力・テスト）が変わるか?
      いいえ → Assume（妥当な既定値で確定し、記録する）
      はい  → 次へ

(3) 既存・規約・文脈から、妥当な既定値を1つに絞れるか?
      はい  → Assume（その既定値で確定し、記録する）
      いいえ → Ask（spec が本質的に分岐している。質問する）
```

判定軸は「気になるか」でも「危険か」でもなく、**「spec が一意に決まるか」**。

---

## 3. Investigate（調べて決める）

人に聞く前に、リポジトリから spec を導出する。

| 知りたいこと | 調査先 |
| --- | --- |
| 既存 API 仕様 | route / controller / schema / test |
| 既存 UI パターン | component / 既存画面 |
| 入出力・データ形式 | model / schema / 既存レスポンス |
| エラー処理 | error class / handler / 既存 test |
| テスト・lint・build 実行方法 | package scripts / CI 設定 / 既存 test |
| 命名規則・依存ライブラリ | 周辺コード / lock file |
| 既存の挙動（削除方式・保存有無など） | 同種機能の既存実装 |

「聞けば分かること」と「読めば分かること」を混同しない。読めば分かるなら聞かない。

---

## 4. Assume（仮定して決める）

導出しきれない点は、最も妥当な spec を **1つ仮定して** 進める。条件:

- 既存コード・規約から自然に推定できる
- 後から簡単に変更できる（可逆）
- 仮定を明示・記録できる

仮定は必ず記録する（レビューできる形に）。

```text
Assumption:
- 未確定だった点:
- 仮定した spec:
- 根拠:
- 影響範囲:
- 変える場合の修正箇所:
```

仮定は隠さない。記録してあれば、人はレビューで覆せる。これが承認待ちで止めるより速い。

---

## 5. 仮決めのデフォルト（1つに絞るときの既定値）

すべて **「最小・可逆・既存に整合」** の方向に倒す。迷ったら、後から一番直しやすい方を選ぶ。

| 不明な点 | 既定の spec |
| --- | --- |
| 実装範囲 | 課題が要求する最小範囲にする |
| 既存仕様との関係 | 後方互換を保つ（既存の呼び出し側を壊さない） |
| 入出力形式 | 既存の同種 API・型に合わせる |
| エラー時の挙動 | 既存のエラーハンドリングに合わせる |
| UI 表現 | 既存デザイン・既存コンポーネントに合わせる |
| 命名 | 既存の命名規則に合わせる |
| テスト方針 | 変更箇所に最も近い既存テストに合わせる |
| 依存ライブラリ | 新規依存を足さない |
| データ保存の有無 | 同種データの既存の扱いに合わせる |
| 削除方式 | 既存の削除方式（論理 / 物理）に合わせる |

---

## 6. Ask（聞いて決める）— 最後の手段

spec が本質的に分岐し、導出も妥当な仮定もできないときだけ質問する。

- `AskUserQuestion` を使い、**最大3問**。
- 自由記述で広く聞かず、**選択肢＋推奨デフォルト** の形にする。
- 質問1つにつき「その答えで spec の何が変わるか」を添える。

**推奨デフォルトを必ず付ける。** 答えが返らなくても、そのデフォルトで Assume に倒して進められる。
これで質問が人待ちのボトルネックにならない。

質問してよい代表項目: 成功条件 / 対象範囲・非対象範囲 / 主要なユーザー操作 /
異常系の期待挙動 / 入出力形式 / データの永続化方針。

優先順位（spec への効き目が大きい順）:

```text
1. 成功条件  2. 対象範囲  3. 主要挙動(入出力)  4. 異常系  5. データ方針  6. UI詳細
```

4問以上要るなら、課題が大きすぎる。質問せず **課題を分割** し、各塊を個別に spec 化する。

---

## 7. 聞かないこと（既存から決める）

フォーマット / import 並び順 / 変数名の細部 / ファイル配置 / テストファイル名 /
UI コンポーネント選択 / エラークラス選択 / ログ形式 / 新規依存の有無。
→ formatter・lint・既存構造・既存設計から一意に決まる。聞かない。

---

## 8. 実装前に必ず出力する（確定版の要件）

```text
- 課題の理解
- 成功条件（テスト可能な形で）
- 対象範囲 / 非対象範囲
- 入力 / 出力
- 主要な異常系の挙動
- Investigate で確定した点（と根拠）
- Assume で確定した点（と根拠）
- Ask する点（あれば。なければ「なし」）
- 実装計画
- テスト方針（成功条件をどう検証するか）
```

これがそのまま「定義済みの要件」になる。

---

## 9. 最終判断基準（一文）

**その点が変わると、成功条件・挙動・入出力・テストのいずれかが変わるなら、
それは spec を確定すべき重要点。既存実装や規約から決まり、後から簡単に直せるなら、
仮定して進めてよい点。人に聞くのは、spec が本質的に分岐し、調べても仮定でも
一つに絞れないときだけ。**
