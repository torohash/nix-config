# domain 値の意味を明示する

## 原則

domain に紐づく値（定数・設定フィールド・境界値・閾値）は、**名前＋型で意味が一意に定まらないなら**、定義箇所に **1行で domain 上の意味**を書く。特に修飾子（max / min / limit / threshold / timeout / retry / interval …）が「**何に**」「**どの単位で**」かかるかを明示する。

目的は「domain をコードから読み取るしかない」状態を無くし、定義を見た瞬間に腹落ちできるようにすること。

## コメントが必須の条件（これに当たるなら書く）

- 修飾子付きで何にかかるか曖昧: `maxCancelFailures`（上限？ 許容失敗数？ 同時数？）
- 単位が名前から不明: ms / s? 件 / 回? bytes / KB? 割合 / 絶対値?
- domain 固有の閾値・ビジネスルール由来の数値・マジックナンバー
- 各値の意味が自明でない enum 的な定数

## 書く内容（最低限）

- その値が domain 上「何を」表すか
- 修飾子が掛かる対象 / 単位
- 理想: 超過・境界時の挙動まで

```ts
// ✗ max が何にかかるか読み手には確定しない
const maxCancelFailures = 3;

// ✓ 1行で確定する
// 1バッチ内で許容するキャンセル失敗の最大件数。超えたらバッチ全体を中断する。
const maxCancelFailures = 3;
```

## 書かなくてよい（ノイズ回避）

- 名前＋型で意味が自明な値
- domain 非依存の技術的に自明な値

「全部にコメント」は逆にノイズ。**曖昧なものだけ**が対象。

## コメント以外でも満たせる（同等の手段）

曖昧さが消えるなら、コメントの代わりにこれでもよい:

- 単位・対象を名前に埋める: `cancelFailureLimitPerBatch`, `requestTimeoutMs`, `maxConcurrentCancels`
- branded / 単位型で型に意味を持たせる: `Milliseconds`, `Count`, `Ratio`

判定基準は「名前 or 型で曖昧さが消えているか」。消えていなければ 1行コメント必須。

## 弾く anti-pattern

- 修飾子付き・単位不明の値を、コメントも命名の工夫もなく定義
- マジックナンバーを裸で埋め込む（意味の説明なし）
- 「なんとなく分かる」名前で放置し、domain 挙動（超過時など）を隠す

## 監査観点（Claude が差分をレビューする時）

- 新規の定数・設定フィールド・境界値に、曖昧なのに解説が無いものはないか
- 修飾子（max / min / limit / timeout …）が何にかかるか、単位が定義箇所で分かるか
- 命名 or 型で曖昧さが消えているなら OK、消えていないなら 1行を要求する
