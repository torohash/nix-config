# work-handoff

context を定期的にリフレッシュしても作業に復帰できるよう、対象プロジェクト直下に
gitignore された単一の引き継ぎ資料 (`WORK_HANDOFF.md`) を作成/更新する。

目的は **技術的な引き継ぎ**: 「このプロジェクトで何をしたいか」「何が完了したか」
「次に何をするか」「注意点」を、後から読む自分（context リフレッシュ後の Claude）が
迷わず作業を再開できる形で短く残すこと。

## 成果ファイル

- パス: project root の `WORK_HANDOFF.md`（既定）。
  - ユーザーが別名を指定した場合はそれを使う。
- gitignore 対象。リポジトリにはコミットしない。
- **毎回全文上書き**（履歴は残さない。常に「今の状態」だけを保持）。
- 言語は日本語。できる限り短く、箇条書き中心。

## 手順

1. **git 情報を自動収集**（読み取りのみ。git project でない場合は branch/差分の記載を省略）:
   - `git rev-parse --abbrev-ref HEAD` … branch
   - `git status --short` … 変更ファイルの概要
   - `git log --oneline -5` … 直近 commit
   - `git diff --stat` … 未コミット差分の概要

2. **git 事実 + 直近の会話文脈を統合**してテンプレートを埋める。
   - git で分かる事実の羅列で終わらせない。会話文脈から「何のために」「次に何を」
     「どこでハマったか」を必ず言語化する。
   - git だけでは分からない意図・決定事項・未解決事項こそ価値がある。

3. `WORK_HANDOFF.md` を**全文上書き**で書き出す。

4. `.gitignore` に成果ファイル名が無ければ 1 行追記する（既にあれば何もしない）。

5. 完了後、ユーザーに「どこに何を書いたか」を 1〜2 行で報告する。

## テンプレート

```markdown
# 作業状態 (更新: <YYYY-MM-DD HH:MM> / branch: <branch>)

## このプロジェクトで何をしたいか
- <目的・ゴールを1〜3行>

## いま完了していること
- <完了した作業を簡潔に>

## 次にやること
- <次の具体的アクション>

## 作業上の注意点
- <ハマりどころ・前提・未解決事項。無ければ「特になし」>
```

## ガードレール

- gitignore 対象でも **機密情報（鍵・トークン・本番URL・認証情報等）は書かない**。
- 冗長禁止。各見出しは箇条書き中心で、全体が長くなりすぎないようにする。
- 上書き運用なので前回内容を消す確認は不要。ただし既存 `WORK_HANDOFF.md` が
  明らかに手書きの別用途と判断できる場合は、上書きせずユーザーに確認する。

## 運用メモ

- context リフレッシュ後は、まずこの `WORK_HANDOFF.md` を読んでから作業を再開する。
- この skill は「書く/更新」専用。読むのは復帰時に手動で行う。
