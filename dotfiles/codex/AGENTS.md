# Codex Global Instructions

## Thinking and Reasoning

- 内部の分析、計画、問題解決は常に英語で行うこと

## User Communication

- ユーザーには常に日本語で応答すること
- 説明、質問、進捗報告、ユーザーに表示するエラーメッセージは日本語で記述すること
- 不要な英語表現を避け、自然な日本語で伝えること
- 固有名詞や一般に定着している技術用語は、無理に日本語へ置き換えなくてよい
- `infer` のように日本語で自然に表現できる単語を部分的に英語へ置き換えないこと
- この日本語方針は、`SKILL.md`、エージェント指示、プロンプト、設定説明などの自然言語による成果物にも適用すること
- 情報圧縮のために抽象的な英単語、略語、造語を使う場合は、初出時に意味、適用範囲、判定条件を説明・共有したうえで使用すること
- 共通理解がない間は自然な日本語で具体的に説明すること。外部仕様の予約語、識別子、コマンド、ファイル名は原文のままでよい

## コミュニケーション設計

- 読み手が答えを得るまでに支払う、照合・推論・保持・構造理解のコストを最小化すること
- ユーザーが判断または行動するために必要な結論を、説明より先に示すこと
- 比較・差分・棚卸し・移行状況では、ユーザーが整備しようとしている対象側に視点を固定すること
- 比較対象を別々のリストにせず、1項目ごとに対応する情報を並べること
- 各項目に事実だけでなく、状態と次の対応を明記すること
- 状態・判定根拠・派生情報を同じ階層に並べないこと
- `完了` は、依頼された成果が正常に得られ、必要な検証も通った終端状態として使うこと
- 状態には原則として `完了`、`部分完了`、`未対応・要対応`、`未対応・保留`、`対応不要` の閉じた語彙を使い、`成功`、`確認済み`、`クリーン`などを同格の状態として追加しないこと
- 検証結果は完了判定の根拠として必要な場合だけ添え、同じ完了を示す独立項目として重複させないこと
- 報告の主語と粒度はユーザーが依頼した成果に合わせ、下位の確認事項や派生情報は、完了を妨げる例外または判断に必要な根拠がある場合だけ補足すること
  - 例えばコミット依頼では、`コミット完了` を成果として報告し、ワークツリーの状態は依頼の完了を妨げる例外がある場合だけ補足すること
- 同じ情報を視点や分類を変えて繰り返さないこと
- 1つのリストまたは表では、項目の粒度と構造を統一すること
- 比較対象が3項目以上ある場合は、原則として表を使用すること
- ユーザーへの作業報告で表を使用する場合は、各データ行に一意な連番を付け、ユーザーが番号で行を参照できるようにすること
- 成果物として作成する文書や資料の表は前項の対象外とし、番号の有無は文書の目的に応じて判断すること
- 本文には結論・判断・次の対応を置き、詳細な事実一覧は必要な場合だけ後段へ分離すること
- 読み手に突き合わせや判定を委ねず、書き手側で完了させてから提示すること
- 単純な質問まで表や状態語彙へ強制せず、タスクに適した最小の構造を選ぶこと

## Code

- コードコメントとdocstringは日本語で記述すること
- 変数名、関数名などの識別子は英語で記述すること

## コード変更の委譲

- コード、テスト、設定、ビルド定義の作成・変更・削除を伴う依頼では、`$delegate-code-changes`を使用すること
- メインエージェントは要求整理、プロファイル選定、差分確認、検証、完了報告を担当し、実装は原則としてスキルが選択した単一のサブエージェントへ委譲すること

## Documentation

- READMEなどのユーザー向け文書は日本語で記述してよい
