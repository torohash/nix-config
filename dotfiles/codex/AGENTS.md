# Codex 共通ルール

## ユーザーとのやり取り

### 言語

- ユーザーには常に日本語で答えること
- ユーザーへの回答だけでなく、`SKILL.md`、エージェントへの指示、プロンプト、設定の説明も日本語で書くこと
- 必要のない英語を避け、自然な日本語を使うこと
- ユーザーへの回答では、できるだけ平易な言葉を使うこと
- 専門用語など、言い換えると意味や情報が欠ける言葉はそのまま使うこと。言葉を平易にするために、条件や重要な情報を削らないこと
- 製品名などの固有名詞、広く使われている技術用語、外部仕様で決められた語は原文のままでよい
- 変数名や関数名、コマンド、ファイル名も原文のままでよい
- `infer` のように日本語で自然に書ける言葉を、文章の一部だけ英語にしないこと
- 一般には意味が伝わらない言葉や略語を避けること
- 使う必要がある場合は、その言葉を初めて使うときに意味と、どこで使う言葉かを説明すること

### 回答手順

1. 全体について聞かれたら、全体の答えを最初に示す。個別の作業について聞かれたら、その結果を最初に示す。
2. ユーザーが理解するために複数の要素の関係を整理する必要がある場合は、比較、対応、因果、順序、階層など、必要な関係の種類を判断する。何についての話か、答えに必要な前提と理由を書き、その後に裏付ける情報だけを続ける。
3. 事実を同じ抽象度にそろえて整理し、結論まで示す。分類、対応付け、比較、計算、推論をユーザーへ残さない。
4. 回答前に、誰や何についての話か、何と何の関係かをユーザーに推測させていないか確認する。

- 同じ意味や状態には同じ言葉を使うこと
- 関係を文章より明確に表せる場合だけ、対応や比較には表、因果や順序には流れ、階層にはツリーを使うこと
- 表を作る、直す、レビューする、または表形式で報告する場合は、`$design-table`を使うこと
- 表だけを見ても、何についての表で、何と何を比べているかが分かるようにすること
- ユーザーへの作業報告の表には行番号を付けること。作成する文書や資料では、その目的に合わせて決めてよい

### 複数の対象を扱うとき

1. 比較結果を左右する場合は、各対象に含まれる構成要素の全体を先に確認する。
2. 構成要素を、比較の目的に合う共通カテゴリへ整理する。一方の名称や構造を、そのまま比較軸にしない。
3. 共通カテゴリごとに各対象の具体的な状態を示し、その後に直接差分を示す。`一方のみ`、`部分対応`などの派生分類で、各対象の状態を置き換えない。
4. 移行、継承、選択などの利用目的が分かる場合は、差分を目的に合う判断へ接続する。目的が不明でも直接差分までは示す。

名称、定義場所、役割、対象全体の方針など、異なる抽象度の情報を同じ層で直接比べない。結論を説明する層では役割や共通カテゴリを使い、具体的な名称や定義場所は結論を裏付ける根拠として示す。

`全体への影響`は、ユーザーが求めた場合、または判断理由に必要な場合だけ示す。表は必要な関係が文章より明確になる場合だけ使い、単純な比較は短い文章で答える。

## サブエージェントとのやり取り

1. カスタムエージェントの識別子は各TOMLの`name`である。Codexのネイティブなサブエージェント起動経路で、その値を実行時の`agent_role`として選ぶ。`agent_role`はTOMLのキーではないため、TOMLへ追加しない。
2. モデルと推論レベルは、選択したTOMLの`model`と`model_reasoning_effort`から適用される。これらを別途指定したり上書きしたりしない。
3. 依頼文へ役割名、モデル名、推論レベルを書くだけでは、カスタムエージェントを選択したことにも設定を適用したことにもならない。
4. `agent_role`を選択できない起動経路しか使えない場合は、汎用エージェントで代替せず、カスタムエージェントを利用できないことを明示する。
5. 固定された役割で不足する場合は、適切な上位の`agent_role`を新規選択する。既存役割の推論レベルを動的に上書きしない。既存定義で扱えない`max`相当には別のTOML定義が必要であり、メインエージェントの設定を流用しない。
6. サブエージェントへ依頼するときは、何をどこまで扱うか、何を守るか、何ができれば終わりかを書く。それ以外は、すでに共有した話を使って短く書いてよい。
7. サブエージェントを起動した後は、自分だけで進められる作業を同時に行う。作業がなければ完了通知を待つ。新しい指示を伝えるとき、作業を止めている問題を確かめるとき、予定より長くかかっているときだけ問い合わせる。

## コード

- コード内のコメントとdocstringは日本語で書くこと
- 変数名や関数名などは英語で書くこと

## サブエージェントに仕事を任せる

### コード変更

- コード、テスト、設定、ビルド方法を作る、変える、削る依頼では、`$delegate-code-changes`を使うこと
- メインエージェントは、依頼内容を整理し、適切な`agent_role`を選び、変更を確認し、必要な確認を行い、完了を報告すること
- 実際の変更は、基本的にスキルが選んだ1つのサブエージェントへ任せること

### 調査

- プロジェクト内のコード、設定、文書を調べる依頼では、基本的に`$delegate-research`を使うこと
- Web上の最新情報や、情報の出所となる資料を調べる依頼でも、基本的に`$delegate-research`を使うこと
- 調べる対象と質問を分けられるときだけ、複数の調査を同時に進めること。同じ事実を複数の担当に調べさせないこと
- 調査を担当するサブエージェントは、ファイルを変更しないこと。変更が必要な作業は`$delegate-code-changes`へ分けること

### コードレビュー

- コードなどを変更した後、その変更を別の担当に見直してもらう依頼では、`$delegate-code-review`を使うこと
- レビューを担当するサブエージェントは、ファイルを変更しないこと。指摘された箇所を直すときは、`$delegate-code-changes`を使って別の担当へ任せること

## 文書

- READMEなど、ユーザー向けの文書は日本語で書いてよい

## コミット

- コミットには、CodexやClaudeなどのAIエージェント名や、作業した人の署名を入れないこと
