# コード変更を委譲する

メインエージェントは要求整理、役割選定、差分確認、検証、完了報告を担当する。実装は原則として選択した単一のカスタムエージェントへ委譲する。

選択対象のCodex識別名（許可一覧の唯一の情報源）: `code-change-advanced`,`code-change-mechanical`,`code-change-standard`

## 役割を選んで起動する

1. ユーザー要求、作業対象へ適用される`AGENTS.md`、既存変更、変更範囲、成果が保証すべき振る舞い、境界条件、許容する仮定、意図的な対象外、完了条件を確認する。
2. 変更量ではなく、必要な判断と設計責任から次の役割を選ぶ。
   - `code-change-mechanical`: 仕様、実装経路、完了条件が明確で、設計判断を要しない定型変更。
   - `code-change-standard`: 通常の機能追加、バグ修正、テスト追加、既存設計内のリファクタリング。
   - `code-change-advanced`: 新たな設計選択、アーキテクチャ変更、移行、複数システム間の変更、高リスク領域。
3. 前掲の許可一覧から選んだ役割名を`~/.codex/agents/*.toml`の`name`と照合する。
4. 利用中のネイティブな起動機能に、カスタムエージェント名を指定する公開された入力がある場合だけ、選んだ`name`を正確に指定して1体起動する。モデル、推論レベル、担当範囲はTOMLから適用し、起動時に上書きしない。ファイル名、表示nickname、追跡用の`task_name`は選択に使わず、`task_name`は役割名と独立した作業名にする。
5. 起動機能に会話履歴の引き継ぎ範囲を指定する公開された入力があれば`none`相当を選び、必要な文脈を委譲契約へ自己完結的に書く。利用中の機能が全履歴の引き継ぎとカスタムエージェント指定の併用を明示的に保証する場合だけ例外を認める。
6. カスタムエージェント名を指定する公開された入力がない場合、または選んだカスタムエージェントを利用できない場合は、built-inや依頼文だけの汎用エージェントで代替せず、委譲できないことを明示する。公開仕様にない引数を作らない。
7. 委譲契約を渡し、起動機能が返したagentまたはthreadの識別子と担当を対応付ける。完了後に差分と検証結果を確認する。

## 委譲契約を渡す

周囲の会話履歴を参照しなくても担当が実行できるように、指示へ次の項目を具体的に記載する。

```text
目的:
変更範囲:
成果が保証すべき振る舞い:
境界条件:
許容する仮定:
意図的な対象外:
守るべき制約:
完了条件:
実行する検証:
変更してはいけないもの:
役割を変更すべき条件:
```

外部から見える成果の意味、正しさ、適用範囲を変える判断を、実装担当に暗黙に補わせない。これらを左右する仕様が確定していない場合は、通常の内部実装の選択と区別してメインエージェントへ返させる。一方、契約済みの振る舞いを変えない定型的な内部実装の選択まで、ユーザー判断として差し戻させない。

同じファイルまたは同じ責任範囲を複数の書き込み担当へ委譲しない。独立した調査、検証、レビューだけを並列化できるが、書き込み担当は常に1体に保つ。

## 完了まで待つ

- 起動したagentまたはthreadの完了通知かterminal stateまで、利用中の待機機能が公開する入力だけを使って待つ。待機時間だけを受け取る場合は識別子を渡さず、待機区間の終了後に一覧・状態照会または`/agent`で対象状態を確認する。対象指定付き待機が公開されている場合だけ、その入力へ識別子を渡す。
- 待機呼び出しや待機区間が通知なしで終了しても、状態が起動準備中、activeまたはrunning相当なら待機を続ける。追加指示、状態照会、中断では、それぞれの操作が対象を指定する公開された入力を持つ場合にだけ記録した識別子を使う。
- 通常の直接起動へ一律の短い制限時間を設けない。特に`high`と`xhigh`は長時間になり得る。`agents.job_max_runtime_seconds`は`spawn_agents_on_csv`専用であり、通常の直接起動の中断判定には使わない。
- 長時間作業では利用者へ進捗を伝える。進捗通知がない場合や長期間状態が変わらない場合は、状態、thread、直近の活動、承認・tool・外部入力待ちの有無を確認し、同じ催促を繰り返さない。
- 利用者の取消、依頼の置換、明示的なterminal failure、または承認・tool・外部入力待ちがなく進行が止まり、具体的な追加情報か1回の進行方向の修正でも再開しないことを確認した場合だけ中断または再選定する。単なる経過時間や短い待機区間の終了では中断しない。
- `completed`相当でも、委譲契約が求める変更と検証結果、および残作業がある場合は未実施範囲が返ったことを確認して成功とする。結果欠落、error、interrupted、not found相当は成功として扱わない。

## 未完了時に再選定する

- 機械的変更の範囲を超えたら`code-change-standard`、既存設計内の判断を超えたら`code-change-advanced`を新たな実装担当として選び直す。既存担当の設定を動的に上書きしない。
- `code-change-advanced`の固定設定でも扱えない`max`相当の作業には、専用のTOML定義が必要である。定義がなければ利用不能とし、メインエージェントの設定を流用しない。
- 不足していた指示や作業中に得られた事実を補うだけなら同じ担当へ追加指示できる。情報、権限、外部状態またはユーザー判断が不足する場合は、役割を上げず不足を解消する。

同じ担当へ情報を追加するときは既存の識別子を使う。別の役割を新規起動する前に、前の担当がterminal stateになったことを確認し、書き込み担当が同時に複数動かないようにする。
