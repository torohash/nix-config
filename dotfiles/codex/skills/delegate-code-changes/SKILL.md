# コード変更を委譲する

メインエージェントは要求整理、役割選定、差分確認、検証、完了報告を担当する。実装は原則として選択した単一のカスタムエージェントへ委譲する。

## 役割を選んで起動する

1. ユーザー要求、作業対象へ適用される`AGENTS.md`、既存変更、変更範囲、成果が保証すべき振る舞い、境界条件、許容する仮定、意図的な対象外、完了条件を確認する。
2. 変更量ではなく、必要な判断と設計責任から次の役割を選ぶ。
   - `code-change-mechanical`（`gpt-5.6-luna` / `medium`）: 仕様、実装経路、完了条件が明確で、設計判断を要しない定型変更。
   - `code-change-standard`（`gpt-5.6-terra` / `high`）: 通常の機能追加、バグ修正、テスト追加、既存設計内のリファクタリング。
   - `code-change-advanced`（`gpt-5.6-sol` / `xhigh`）: 新たな設計選択、アーキテクチャ変更、移行、複数システム間の変更、高リスク領域。
3. カスタムエージェントの識別子は各TOMLの`name`である。Codexのネイティブなサブエージェント起動経路で、選んだ`name`を実行時の`agent_role`として指定する。`agent_role`はTOMLのキーではないため、TOMLへ追加しない。
4. モデルと推論レベルは選んだTOMLの`model`と`model_reasoning_effort`から適用される。これらを別途指定したり上書きしたりしない。
5. 依頼文へ役割名、モデル名、推論レベルを書くだけでは、カスタムエージェントを選択したことにも設定を適用したことにもならない。
6. `agent_role`を選択できない起動経路しか使えない場合は、汎用エージェントで代替せず、カスタムエージェントを利用できないことを明示する。
7. 委譲契約を渡して書き込み担当を1体だけ起動し、完了後に差分と検証結果を確認する。

## 委譲契約を渡す

指示に次の項目を具体的に記載する。

```text
目的:
変更範囲:
成果が保証すべき振る舞い:
境界条件:
許容する仮定:
意図的な対象外:
守るべき制約:
完了条件:
実行する検証:
変更してはいけないもの:
役割を変更すべき条件:
```

外部から見える成果の意味、正しさ、適用範囲を変える判断を、実装担当に暗黙に補わせない。これらを左右する仕様が確定していない場合は、通常の内部実装の選択と区別してメインエージェントへ返させる。一方、契約済みの振る舞いを変えない定型的な内部実装の選択まで、ユーザー判断として差し戻させない。

同じファイルまたは同じ責任範囲を複数の書き込み担当へ委譲しない。独立した調査、検証、レビューだけを並列化できるが、書き込み担当は常に1体に保つ。

## 未完了時に再選定する

- 機械的変更の範囲を超えたら`code-change-standard`、既存設計内の判断を超えたら`code-change-advanced`を新たな`agent_role`として選び直す。既存役割の推論レベルは動的に上書きしない。
- `code-change-advanced`の固定設定でも扱えない`max`相当の作業には、専用のTOML定義が必要である。定義がなければ利用不能とし、メインエージェントの設定を流用しない。
- 不足していた指示や作業中に得られた事実を補うだけなら同じ担当へ追加指示できる。情報、権限、外部状態またはユーザー判断が不足する場合は、役割を上げず不足を解消する。

新規起動前に前の担当を終了させ、書き込み担当が同時に複数動かないようにする。
