# コード変更を委譲する

メインエージェントは要求整理、プロファイル選定、委譲、差分確認、検証、完了報告を担当する。実装は原則として選択したサブエージェントへ委譲する。

## 選定手順

1. ユーザー要求、作業対象へ適用される`AGENTS.md`、既存変更、変更範囲、完了条件を確認する。
2. 作業の性質と判断責任からモデルを選ぶ。行数やファイル数だけで選ばない。
3. 依存関係、リスク、曖昧さ、検証負荷から推論レベルを選ぶ。最低でも`medium`を使う。
4. 対応するプロファイルを選び、書き込み担当サブエージェントを1体だけ起動する。
5. サブエージェントの差分と検証結果を確認する。不足があれば同じサブエージェントへ追加指示するか、上位プロファイルへ再委譲する。
6. メインエージェントが必要な最終検証を行い、結果を報告する。

## モデルを選ぶ

| モデル | 選択条件 |
|---|---|
| Luna | 仕様、変更内容、実装経路、完了条件が完全に明確な機械的作業。コメント追加、指示済みの定数化、厳密に指定された名前変更、定型変換などに使う。 |
| Terra | 通常の機能追加、バグ修正、テスト追加、既存設計内のリファクタリングなど、限定的な実装判断が必要な作業に使う。 |
| Sol | 要求が曖昧な作業、設計選択、アーキテクチャ変更、大規模リファクタリング、移行、複数システム間の変更、セキュリティ・認証・並行処理・データ整合性など高リスクな作業に使う。 |

変更量が多くても機械的ならLunaを選べる。変更量が少なくても設計判断や高いリスクがあればTerraまたはSolを選ぶ。

## 推論レベルを選ぶ

| レベル | 選択条件 |
|---|---|
| `medium` | 実装経路がほぼ一意で、影響範囲と検証方法が明確である。 |
| `high` | 複数の実装候補、非自明な制御・データフロー、互換性、境界条件、原因調査がある。 |
| `xhigh` | 複数コンポーネントの相互作用、設計上のトレードオフ、広い影響範囲、非同期処理、移行、検証上の不確実性がある。 |
| `max` | 分割しにくい最難関問題で、失敗時の影響または不可逆性が大きい、あるいは過去の修正が繰り返し失敗している。Solだけで使う。 |

次の昇格規則を必ず適用する。

- Lunaで`xhigh`以上が必要なら、作業の性質を再評価してTerraまたはSolへ昇格する。
- Terraで`max`または設計責任が必要ならSolへ昇格する。
- Solは原則として`high`以上を使う。Sol相当で`medium`と判断した場合も`code-sol-high`を使う。

## プロファイルを選ぶ

利用可能なプロファイルを次に限定する。

| モデル | `medium` | `high` | `xhigh` | `max` |
|---|---|---|---|---|
| Luna | `code-luna-medium` | `code-luna-high` | 上位モデルへ昇格 | 上位モデルへ昇格 |
| Terra | `code-terra-medium` | `code-terra-high` | `code-terra-xhigh` | Solへ昇格 |
| Sol | `code-sol-high` | `code-sol-high` | `code-sol-xhigh` | `code-sol-max` |

## 委譲する

サブエージェントへの指示に次の契約を明記する。

```text
目的:
変更範囲:
守るべき制約:
完了条件:
実行する検証:
変更してはいけないもの:
上位プロファイルへ昇格すべき条件:
```

同じファイルまたは同じ責任範囲を複数の書き込み担当へ委譲しない。独立した調査、検証、レビューだけを並列化できるが、書き込み担当は常に1体に保つ。

サブエージェントが割り当てられた難度または責任範囲を超える問題を検知した場合は、推測で作業を拡張させない。編集を止め、理由、推奨する上位プロファイル、実施済み作業、未実施作業を返させる。メインエージェントは内容を確認し、適切な上位プロファイルへ再委譲する。
