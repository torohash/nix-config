# コードレビューを委譲する

実装担当とは別の`code-review`を、必ず読み取り専用で起動する。レビュー担当に修正を実装させない。

## 起動して結果を受け取る

1. レビュー対象の差分、要求が約束する成果、成果を成立させる具体的な処理経路と境界条件、適用される`AGENTS.md`、実施済みの検証、既知の制約を整理する。
2. `code-review`をCodexのサブエージェント機能で起動する。モデル、推論レベル、担当範囲はエージェント定義から適用し、起動時に上書きしない。
3. `code-review`を利用できない場合は汎用エージェントで代替せず、委譲できないことを明示する。
4. 読み取り専用で作業し、編集、コミット、追加のサブエージェント起動を行わないよう明記する。
5. 要求が約束する成果と、具体的な処理経路、境界条件、テストの対応を確認させ、必要な振る舞いが暗黙に省略されていないか検査させる。
6. 問題を重大度順に報告させ、それぞれに根拠、場所、具体的な影響、修正方向を対応付けさせる。期待する振る舞いを確定できない場合は不具合と断定させず、仕様上の未決定として返させる。
7. 問題がない場合はその旨と、未検証範囲または残る危険を返させる。修正が必要なら`$delegate-code-changes`を使って別途委譲する。

## 固定設定で扱えない場合

`code-review`の固定設定でも扱えない`max`相当のレビューには、専用のTOML定義が必要である。定義がなければ利用不能とし、既存役割の推論レベルを上書きしたり、メインエージェントの設定を流用したりしない。

差分、仕様、検証結果、再現条件などの情報が不足する場合は役割を変えず、不足を解消する。不足していた情報を補うだけなら同じ担当へ追加指示できる。
