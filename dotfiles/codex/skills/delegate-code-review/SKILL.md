# コードレビューを委譲する

実装担当とは別の`code-review`を、必ず読み取り専用で起動する。レビュー担当に修正を実装させない。

選択対象のCodex識別名（許可一覧の唯一の情報源）: `code-review`

## 起動して結果を受け取る

1. レビュー対象の差分、要求が約束する成果、成果を成立させる具体的な処理経路と境界条件、適用される`AGENTS.md`、実施済みの検証、既知の制約を整理する。
2. Codex識別名`code-review`を`~/.codex/agents/code-review.toml`の`name`と照合する。
3. 利用中のネイティブな起動機能に、カスタムエージェント名を指定する公開された入力がある場合だけ、`code-review`を正確に指定して起動する。モデル、推論レベル、担当範囲、`sandbox_mode`はTOMLから適用し、起動時に上書きしない。ファイル名、表示nickname、追跡用の`task_name`は選択に使わず、`task_name`は役割名と独立した作業名にする。
4. 起動機能に会話履歴の引き継ぎ範囲を指定する公開された入力があれば`none`相当を選び、必要な文脈をレビュー指示へ自己完結的に書く。利用中の機能が全履歴の引き継ぎとカスタムエージェント指定の併用を明示的に保証する場合だけ例外を認める。
5. カスタムエージェント名を指定する公開された入力がない場合、または`code-review`を利用できない場合は、built-inや依頼文だけの汎用エージェントで代替せず、委譲できないことを明示する。公開仕様にない引数を作らない。
6. TOMLの`sandbox_mode = "read-only"`に加え、読み取り専用で作業し、編集、コミット、追加のサブエージェント起動を行わないよう指示へ明記する。親turnのlive permission overrideが優先され得るため、禁止指示を省略しない。
7. 起動機能が返したagentまたはthreadの識別子と`code-review`を対応付ける。
8. 要求が約束する成果と、具体的な処理経路、境界条件、テストの対応を確認させ、必要な振る舞いが暗黙に省略されていないか検査させる。
9. 問題を重大度順に報告させ、それぞれに根拠、場所、具体的な影響、修正方向を対応付けさせる。期待する振る舞いを確定できない場合は不具合と断定させず、仕様上の未決定として返させる。
10. 問題がない場合はその旨と、未検証範囲または残る危険を返させる。修正が必要なら`$delegate-code-changes`を使って別途委譲する。

## 完了まで待つ

- 起動したagentまたはthreadの完了通知かterminal stateまで、利用中の待機機能が公開する入力だけを使って待つ。待機時間だけを受け取る場合は識別子を渡さず、待機区間の終了後に一覧・状態照会または`/agent`で対象状態を確認する。対象指定付き待機が公開されている場合だけ、その入力へ識別子を渡す。
- 待機呼び出しや待機区間が通知なしで終了しても、状態が起動準備中、activeまたはrunning相当なら待機を続ける。追加指示、状態照会、中断では、それぞれの操作が対象を指定する公開された入力を持つ場合にだけ記録した識別子を使う。
- `code-review`は`xhigh`で長時間になり得るため、通常の直接起動へ一律の短い制限時間を設けない。`agents.job_max_runtime_seconds`は`spawn_agents_on_csv`専用であり、通常の直接起動の中断判定には使わない。
- 長時間レビューでは利用者へ進捗を伝える。進捗通知がない場合や長期間状態が変わらない場合は、状態、thread、直近の活動、承認・tool・外部入力待ちの有無を確認し、同じ催促を繰り返さない。
- 利用者の取消、依頼の置換、明示的なterminal failure、または承認・tool・外部入力待ちがなく進行が止まり、具体的な追加情報か1回の進行方向の修正でも再開しないことを確認した場合だけ中断または再選定する。単なる経過時間や短い待機区間の終了では中断しない。
- `completed`相当でも、重大度、根拠、場所、影響、修正方向、または問題なしの結論と未検証範囲が返ったことを確認して成功とする。結果欠落、error、interrupted、not found相当は成功として扱わない。

## 固定設定で扱えない場合

`code-review`の固定設定でも扱えない`max`相当のレビューには、専用のTOML定義が必要である。定義がなければ利用不能とし、既存役割の推論レベルを上書きしたり、メインエージェントの設定を流用したりしない。

差分、仕様、検証結果、再現条件などの情報が不足する場合は役割を変えず、不足を解消する。不足していた情報を補うだけなら同じ担当へ追加指示できる。

同じ担当へ情報を追加するときは既存の識別子を使う。別の役割が必要になった場合は、旧担当がterminal stateになったことを確認してから適切なSkillで再選定する。
