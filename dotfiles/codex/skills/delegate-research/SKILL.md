# 調査を委譲する

調査対象をプロジェクト内またはWebに分け、質問に答えるために必要な範囲だけを読み取り専用担当へ委譲する。

## 役割を選んで起動する

1. 調査によって答える問い、問いの抽象度、決めたいこと、必要な根拠、最新性の要否、調査禁止範囲を明確にする。
2. ローカルのコード、設定、テスト、文書を根拠にするならプロジェクト内調査を選ぶ。公開情報、現行仕様、更新状況を根拠にするならWeb調査を選ぶ。両方が必要なら責任範囲を分ける。
3. 対象の明確さ、横断範囲、資料間の矛盾、誤りの影響から次の役割を選ぶ。
   - `project-research-lookup`（`gpt-5.6-luna` / `medium`）: 対象と質問が明確な限定検索。
   - `project-research-synthesis`（`gpt-5.6-terra` / `high`）: 複数のファイルや処理経路を横断する関係・原因候補・影響範囲の統合。
   - `project-research-deep`（`gpt-5.6-sol` / `xhigh`）: 全体設計、難しい原因、広範な影響の調査。
   - `web-research-lookup`（`gpt-5.6-luna` / `medium`）: 参照先がほぼ明確な最新情報と根拠URLの確認。
   - `web-research-synthesis`（`gpt-5.6-terra` / `high`）: 複数の一次資料の比較、日付・対象範囲・矛盾の確認と統合。
   - `web-research-deep`（`gpt-5.6-sol` / `xhigh`）: 不完全・対立する資料、専門性と探索範囲の広い調査、高リスクな判断。
4. カスタムエージェントの識別子は各TOMLの`name`である。Codexのネイティブなサブエージェント起動経路で、選んだ`name`を実行時の`agent_role`として指定する。`agent_role`はTOMLのキーではないため、TOMLへ追加しない。
5. モデルと推論レベルは選んだTOMLの`model`と`model_reasoning_effort`から適用される。これらを別途指定したり上書きしたりしない。
6. 依頼文へ役割名、モデル名、推論レベルを書くだけでは、カスタムエージェントを選択したことにも設定を適用したことにもならない。
7. `agent_role`を選択できない起動経路しか使えない場合は、汎用エージェントで代替せず、カスタムエージェントを利用できないことを明示する。
8. 読み取り専用、編集・コミット・追加のサブエージェント起動禁止を明記して起動する。

## 調査契約を渡す

調査担当へ次の項目を渡す。

```text
調査で答える問い:
問いと同じ抽象度で必要な結論:
事実から導出する必要がある関係:
必要な根拠と判断基準:
確認する範囲:
確認しない範囲:
```

比較、原因、影響範囲などを求める調査では、個別事実の列挙を答えの代わりにせず、指定した関係を根拠から導出させる。具体的な事実や例には、それが支える上位の結論を対応付けさせる。

プロジェクト内調査には、問いと同じ抽象度の結論、根拠となるファイル位置と該当内容、確認済み範囲、未確認範囲を返させる。確定事項と推定を区別させる。

Web調査には、依頼時点の最新性を確認させ、可能な限り一次資料を優先させる。結論、事実を直接支える根拠URL、公開日または更新日、資料の適用範囲、残る矛盾を返させる。

独立していて結論の重複しない調査だけを並列化する。異なる担当へ同じ事実を重複して調べさせない。メインエージェントは複数の結果を並べるだけで終えず、ユーザーが求めた問い、抽象度、前提に合わせて必要な関係を導出し、最終回答へ統合する責任を持つ。

## 未完了時に再選定する

- 限定検索から複数根拠の統合が必要になったら同じ調査領域の`synthesis`、全体設計、難しい原因、高リスク判断が必要になったら`deep`を新たな`agent_role`として選び直す。既存役割の推論レベルは動的に上書きしない。
- `deep`の固定設定でも扱えない`max`相当の調査には、専用のTOML定義が必要である。定義がなければ利用不能とし、メインエージェントの設定を流用しない。
- 不足していた質問や探索対象を補うだけなら同じ担当へ追加指示できる。情報、権限、外部状態またはユーザー判断が不足する場合は、役割を上げず不足を解消する。
