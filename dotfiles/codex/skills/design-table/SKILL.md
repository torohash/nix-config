# 問いとデータ関係から表を設計する

## 手順

1. 読み手が表から確認、判断、実行することを一文で定める。依頼に複数の目的がある場合は、同じ行と列で無理なく表せるかを確認する。
2. 元データの関係を特定する。対象の記録、属性、時間、集計、二軸の交差、順序、階層、対象間の対応、比較など、問いに必要な関係を選ぶ。
3. 1行が表す単位を一文で定める。行ごとに対象の種類や粒度が変わる場合は、表を分割する。
4. 各列が表す意味を定める。同じ列では、意味、型、単位、基準時点を統一する。見出しだけで確定しない場合は、対象、方式、単位、時点、版などを列またはセルで明示する。
5. 対象から得た元状態と、そこから導いた差分、判断、作業、進捗を別の列または別表として扱う。派生した分類や判断で元状態を置き換えない。移行では、旧側の現行実装、新側の目標設計、旧現行から新目標への設計差分、設計上必要な変更を分ける。実施状況も扱う場合は、新側の現在実装、現在実装から目標設計までの残作業、進捗も分け、目標設計から現在実装を推定しない。
6. 関係が文章より明確になる場合だけ表を使う。長文や複数の関係を一つのセルへ詰める場合、表を分割するか、短文、一覧、フロー、タイムライン、グラフ、地図、ツリーへ変える。
7. 該当する参照を読み、問いに合うパターンを選ぶ。ひな型をそのまま転用せず、行と列の意味から必要な項目だけを採用する。

## 参照の選び方

- 一覧、属性、時系列、集計、二軸行列、ランキング、状態・進捗、階層、対応、比較、工程、または表以外の表現を扱う場合は、[一般的な表のパターン](references/general-patterns.md)を読む。
- 移行、要件追跡、互換性、テスト、意思決定、責任分担、リスク・課題、データ辞書を扱う場合は、[ソフトウェア開発の表のパターン](references/engineering-patterns.md)を読む。
- 一般的な関係とソフトウェア開発上の判断が混在する場合は、両方を読む。

## 状態の表し方

- 空欄に意味を持たせない。
- `なし`は確認した結果として値や要素が存在しない状態、`不明`は根拠から確定できない状態、`未取得`は取得対象だがまだ取得していない状態として区別する。
- `対象外`はその行や列を適用しない状態、`未着手`は予定した作業をまだ開始していない進捗として区別する。
- `必須`、`任意`、`対応済み`などの語だけで意味を確定させない。何が、誰に、どの時点または版で、どの方式により適用されるかを、判断に必要な粒度で示す。

## 完了条件

- 表の利用目的と、1行が表す単位を説明できる。
- 各列が一貫した意味、型、単位、基準時点を持つ。
- 元状態、差分、判断、設計上必要な変更、残作業、進捗を混同していない。
- 移行の設計計画では、旧側の現行実装、新側の目標設計、設計差分、設計上必要な変更を区別している。
- 移行の進捗を扱う場合は、新側の現在実装、現在実装から目標設計までの残作業、進捗を区別し、新側の現在実装が確認できなければ`未取得`または`不明`としている。
- 複数対象を比較する場合、各対象の具体的な状態を残し、その後に直接差分を示している。
- `なし`、`不明`、`未取得`、`対象外`、`未着手`を空欄や同じ状態で兼用していない。
- 表を分割すべき関係や、表より適した別表現を一つの表へ押し込んでいない。
